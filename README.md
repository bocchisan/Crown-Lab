# Crown Lab

Ручная лаборатория для проверки всех четырёх игр Crown на **devnet**: страница с кнопками, где видно, как двигаются деньги на кошельках, как канистры принимают голоса и выносят вердикты, и как книга репутации это отражает.

Это **тестовый инструмент, а не продукт**. Он живёт отдельно от ядра сознательно: `Crown-Core` не держит ни игр, ни фронта (см. его `CLAUDE.md`).

```
npm install
./scripts/lab-up.sh     # одна реплика: sol_rpc, crown-index, четыре игры
npm run dev             # открыть напечатанный URL
```

## Что она умеет

| Панель | Сценарии |
|---|---|
| **Участники** | подключить Phantom/Solflare, создать burner-ключи, импортировать существующий ключ, профинансировать любого из любого (SOL + USDC); балансы и **репутация каждого участника** к выбранному стримеру (зелёная — хватает на голос, ≥ 100000) |
| **Ядро** | прямой донат через сплиттер (весь gross, комиссии нет), будильник `ingest_hint` (приближает чтение чейна), чтение книги `get_reputation`, счётчик аномалий, курсор, версия `reduce` |
| **Задания** | эскроу → register → accept/decline → ready → голос → вердикт → claim(0/1); refund() мимо игры; operator-refund; состояние и голоса с весами из сертифицированного ответа |
| **Сбор** | коллекция со своим резолвером → вклады от нескольких участников → ready → голос → один вердикт → claim каждого вклада; отмена получателем; operator-refund; refund() |
| **Аукцион** | аукцион → лоты (резолвер per-lot из [lot_id]) → вклады с живым чтением чейна при регистрации → accept → финал по таймеру → ready → голос → вердикт → claim; возвраты полотово и повкладово; operator-гашение |
| **Подписка** | поток на N кусков → выпуск куска по расписанию → отмена по подписи донора → refund() после `RELEASE_MARGIN` |

## Как устроено доверие к этому коду

Лаборатория — **четвёртая** реализация байтовых форматов Crown (после задеплоенных программ, `crown-salt` и канистр). Копия, разошедшаяся на байт, молча адресует эскроу, которых не существует, поэтому форматы здесь не пишутся «на глаз»:

- `npm test` сверяет соли с векторами фабрики, сообщения — с юнит-тестами канистр, дискриминаторы пересчитывает из `sha256("global:<name>")`, а списки аккаунтов — с `Accounts`-структурами программ;
- сильнейшая проверка — офлайн: в `test/escrow.test.ts` лежат **реальные devnet-эскроу**, и соль пересчитывается из их же полей, воспроизводя те байты, которые записала задеплоенная программа;
- `npm run lint` держит `vectors/*.json` побайтово равными эталону в `Crown-Factory`;
- `npx tsx scripts/smoke.ts` (и `smoke-funding.ts`, `smoke-subscription.ts`) — сквозные headless-прогоны трёх игр против живого devnet и живых канистр, теми же модулями, что и страница;
- `npx tsx scripts/page-check.ts` — грузит страницу настоящим chromium и падает на любой ошибке консоли (сборка и тесты этого не ловят: код может собираться при мёртвой странице). `SEED=1` подсевает ключи e2e и читает таблицу обратно — проверка балансов и репутации через браузер.

## Деньги на devnet

- **Донор** должен держать devnet USDC (`faucet.circle.com`) и немного SOL. Проще всего импортировать существующий ключ e2e (`~/.cache/crown-e2e/donor.json`) кнопкой «импортировать» — вставляется как JSON-массив.
- Остальные роли удобнее держать burner-ключами и финансировать кнопкой «перевести»: они подписывают сообщения канистрам и платят ренту.
- Каждый эскроу стоит ~4M лампортов ренты (возвращается при закрытии), ATA получателя ~2M (не возвращается) — поэтому не плоди эскроу пачками.

## Реплика: почему без `--clean`

Резолвер игры — это threshold-ключ канистры, а на локальной реплике он выводится из canister id. **Снести реплику, пока на devnet живут эскроу, — значит потерять единственный ключ, который может их рассчитать или отменить.** Деньги после этого достаются только через `refund()` после дедлайна (у подписки — через `RELEASE_MARGIN = 900 с`).

Поэтому `lab-up.sh` переиспользует уже поднятую реплику и доливает только недостающее. `WIPE=1 ./scripts/lab-up.sh` — только когда живых эскроу нет.

Все шесть канистр стоят на **одной** реплике (у игр в их репозиториях каждая поднимает свою `--clean`, поэтому втроём они там не живут).

**Если реплику всё же снесло** (падение машины — уже случалось), это не приговор: локальная реплика раздаёт id строго по порядку, поэтому пересоздание тех же канистр в том же порядке возвращает те же id, а с ними те же threshold-ключи и резолверы. Ради этого в `dfx.json` живут `pad1..pad4`: на этой общей реплике четыре id заняла более ранняя сессия ещё до канистр лаборатории, и пустышки сжигают их, удерживая последовательность. Их создают, но никогда не устанавливают.

`lab-up.sh` сверяет полученные id с `.dfx/local/canister-ids.expected` и **останавливается**, если хоть один разъехался: с новым id резолвер — другой ключ, и живые эскроу можно только рефандить после дедлайна. Это ошибка, а не предупреждение.

**Не запускай родные `scripts/e2e-devnet.sh` игр, пока на devnet есть живые эскроу.** Каждый из них начинается с `dfx start --clean`, а локальная сеть общая — снос убьёт threshold-ключи, то есть резолверы этих эскроу. Те же сценарии здесь прогоняются скриптами `smoke*.ts` против уже поднятой реплики; родные e2e игр — когда живых эскроу не осталось.

## Границы

- Страница — недоверенный клиент, как и любой другой: она ничего не решает, только декларирует поля рождения и носит подписи. Неверная декларация выводит адрес, где эскроу не существует.
- Форма `payout-table` здесь не реализована: ни одна игра её не использует, а код без потребителя не нужен.
- Профиль только `testnet`/`local` (devnet). Mainnet тут делать нечего.
